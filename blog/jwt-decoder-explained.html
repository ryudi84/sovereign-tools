JWT Token Decoder: How to Debug JWT Tokens

Published Feb 22, 2026 · 5 min read

JSON Web Tokens (JWTs) are everywhere in modern authentication — from OAuth flows to API authorization. But when something goes wrong, debugging a JWT can feel like deciphering hieroglyphics. This guide breaks down JWT structure, shows you how to decode tokens safely, and covers the most common pitfalls.

What Is a JWT?

A JWT is a compact, URL-safe token format defined in RFC 7519. It consists of three Base64URL-encoded parts separated by dots:

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

Each part has a specific purpose:

  • Header — Algorithm and token type
  • Payload — Claims (the actual data)
  • Signature — Verification hash

Decoding the Header

The header is a Base64URL-encoded JSON object that tells the server how to verify the token:

{
  "alg": "HS256",
  "typ": "JWT"
}

Common algorithms:

  • HS256 — HMAC with SHA-256 (symmetric, shared secret)
  • RS256 — RSA with SHA-256 (asymmetric, public/private key)
  • ES256 — ECDSA with SHA-256 (asymmetric, smaller keys)
  • none — No signature (⚠️ never accept in production!)

Decoding the Payload

The payload contains claims — statements about the user and metadata:

{
  "sub": "1234567890",
  "name": "John Doe",
  "email": "john@example.com",
  "role": "admin",
  "iat": 1516239022,
  "exp": 1516242622
}

Standard Claims (Registered)

  • sub — Subject (user ID)
  • iss — Issuer (who created the token)
  • aud — Audience (intended recipient)
  • exp — Expiration time (Unix timestamp)
  • iat — Issued at
  • nbf — Not before (token isn't valid before this time)
  • jti — JWT ID (unique identifier for the token)

⚠️ Important: The payload is encoded, not encrypted. Anyone with the token can read the claims. Never store secrets, passwords, or sensitive PII in JWT payloads.

How to Decode a JWT in JavaScript

You don't need a library. The payload is just Base64URL-encoded JSON:

function decodeJWT(token) {
  const parts = token.split('.');
  if (parts.length !== 3) throw new Error('Invalid JWT');

  const decode = (str) => {
    // Base64URL → Base64 → decode
    const base64 = str.replace(/-/g, '+').replace(/_/g, '/');
    return JSON.parse(atob(base64));
  };

  return {
    header: decode(parts[0]),
    payload: decode(parts[1]),
    signature: parts[2]
  };
}

Common JWT Debugging Scenarios

1. "Token Expired" (401 Unauthorized)

Check the exp claim. It's a Unix timestamp — convert it to see if the token has expired:

new Date(payload.exp * 1000).toISOString()
// → "2026-02-22T15:30:22.000Z"

If your server and client clocks are out of sync, even a valid token can appear expired. Most libraries allow a small clock skew tolerance (30-60 seconds).

2. "Invalid Signature"

This means the token was tampered with, or the server is using a different secret/key than expected. Common causes:

  • Different secret keys between services
  • Rotating keys without updating all services
  • Using HS256 when the server expects RS256
  • Token was modified in transit (check middleware)

3. "Token Not Yet Valid"

The nbf (not before) claim is in the future. Usually a clock sync issue.

4. Wrong Audience or Issuer

If you're using microservices, tokens issued for Service A might be rejected by Service B because the aud claim doesn't match.

JWT Security Best Practices

  • Always verify the signature — never trust a decoded JWT without verification
  • Reject alg: "none" — a classic attack vector. Attackers strip the signature and set algorithm to "none"
  • Use short expiration times — 15-60 minutes for access tokens, use refresh tokens for persistence
  • Store tokens securely — HttpOnly cookies > localStorage (XSS protection)
  • Validate all claims — check iss, aud, exp, not just the signature
  • Use asymmetric algorithms (RS256/ES256) — allows public key verification without sharing secrets
  • Implement token revocation — JWTs are stateless, so consider a deny-list for compromised tokens

JWT vs Session Cookies: When to Use What

Use JWTs when: You need stateless authentication across microservices, mobile apps, or third-party integrations.

Use session cookies when: You have a single server, need easy revocation, or want simpler security. Sessions are stored server-side, giving you full control.

JWTs aren't always the right choice. They add complexity and can't be easily revoked. Choose based on your architecture, not hype.
